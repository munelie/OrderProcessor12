1. В исходном коде использовалось число 100 в условии отправки email, значение которого никак не объяснялось. Это затрудняло понимание кода. В результате рефакторинга число перенесено в константу EmailThreshold, что упростило возможные изменения.
2. Конструктор класса не проверял переданные зависимости (IDatabase и IEmailService) на null. Это могло привести к ошибкам в процессе выполнения, а не при создании объекта, что усложняло диагностику проблем. В отрефакторенном коде добавлена проверка с использованием оператора ?? и выбрасыванием ArgumentNullException, что позволяет увидеть ошибку на этапе создания объекта.
3. Свойство order.IsProcessed = true устанавливалось в двух разных местах: внутри блока отправки email и после него. Это создавало дублирование и риск того, что при изменении логики одно из мест может быть пропущено. В отрефакторенном коде установка флага обработки вынесена в отдельный метод MarkOrderAsProcessed, который вызывается один раз для всех успешно обработанных заказов.
4. Метод ProcessOrder был слишком большим и делал сразу несколько операций: валидацию, проверку суммы, подключение к БД, сохранение, отправку email, обработку ошибок. Это затрудняло чтение, тестирование и поддержку кода. В отрефакторенном коде метод разбит на маленькие методы с понятными названиями: ValidateOrder, IsOrderAmountValid, EnsureDatabaseConnection, TryProcessOrder, каждый из которых отвечает за одну конкретную операцию.
5. Код содержал неявные операции. В отрефакторенном коде все методы получили понятные названия.
